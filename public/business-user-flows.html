At a glance

  • Visitor — brochure home + KPI strip, browse, PDP, land request, public services marketing, login/register.
  • Buyer — /dashboard; on /properties uses Book visit, shortlist (wishlist), and Enquire (WhatsApp). Saved listings surface to sellers as Interested counts.
  • Seller — /seller (max two active listings); may also use /dashboard as a buyer.
  • Admin / staff — /admin ops hub (staff: no Users tab).

Visitors No account

No login required unless the user wants dashboards or seller tools.

  1. / — storytelling, hero, KPI strip below hero (five public metrics + optional enquiry chip from /api/platform-stats; paddy + turmeric styling). Includes Sample Documents (Sale Deed, Lease, Pahani guide, Registration) in EN & Telugu. Signed-in users are redirected to their role hub (/dashboard, /seller, or /admin). Open /?stay=1 (or ?browse=1) to see the full marketing homepage while logged in.
  2. /properties — filters: State → District → Mandal, soil, land type, acres, price. Cards: Book visit → listing with booking; Add to shortlist / On shortlist (wishlist, max 8; login prompt if anonymous); Enquire opens WhatsApp with listing context — copy-led CTAs (no emoji icons). Optional Road chip when road access is set.
  3. /properties/[id] — PDP; booking, shortlist, WhatsApp helpers. Appointment “Notify Me” if no suitable slot.
  4. /buyer-request — structured requirements; guest or signed-in.
  5. /services — Phase II marketing + booking dialogs (6 services). Signed-in buyers (not seller/admin/staff) are redirected to /dashboard?services=1 (embedded services panel). Anonymous and other roles stay on public /services.
  6. /auth/* — register (buyer/seller) or ?flow=seller shortcut; callback honours homePathForRole and optional ?next=.

Buyers Role: buyer

The buyer dashboard (/dashboard) is the CRM-style view for the buyer herself — operations still control statuses in /admin.

  1. Register at /auth/register (choose Buyer) or sign in at /auth/login.
  2. After email verification (if required) → default home /dashboard.
  3. Overview — account summary, quick call/browse/post-request actions, “journey” strip (registered → enquired → site visit → booked).
  4. Enquiries — threads/messages tied to listings; statuses pending, contacted, visited, booked, closed.
  5. Plot interests — specific plots earmarked inside layouts.
  6. Land requests — outbound brief from /buyer-request; statuses open, in_progress, matched, closed; Edit while open → /buyer-request/[id]/edit.
  7. Continue shopping /properties; listing CTAs funnel into the dashboards above.

Ops updates enquiry/request/plot rows from the hub — buyers mostly read-only except forms they own.

Sellers Role: seller

Sellers manage listings from the seller hub (/seller); SDV staff approve before public visibility. Sellers can also visit /dashboard to browse and enquire as a buyer — both roles work on the same account.

  1. List your land → /auth/register?flow=seller (skips buyer/seller picker).
  2. Eligibility check blocks Poramboke, Forest ceiling land, assigned/inam types, civil dispute land, etc.
  3. Credentials + email verification → dashboard /seller.
  4. Quota — max two active listings counting pending + approved simultaneously.
  5. /seller/property/new wizard — Step 0 now captures Relation to Owner (Self/Wife/Son/Daughter etc.); Step 1 captures Sale Intent (Urgent Sale · Ready to Sell · Interested/Open). Media uploads required before submit.
  6. Approved listings link out to live PDP; views and Interested count (from saved buyers) aggregate on each card.
  7. Appointments tab lists site visits referencing seller inventory.
  8. Nav shows both My Listings and Browse & Buy links. A banner on /dashboard reminds sellers they are in buyer mode.

Admin vs staff Roles

Admin and staff both use /admin. Staff accounts omit the Users tab; only admins can list org users and PATCH roles via /api/admin/users.

  1. Buyer/seller JWTs hitting /admin bounce to dashboards (middleware).
  2. Everyone with ops access triages enquiries, plots, appointments, buyer requests, service bookings, listings, flags.

Operations hub — tabs /admin

  • Enquiries — property lead inbox; statuses pending → contacted → visited → booked/closed.
  • Users — (admin-only) directory + role tooling.
  • Plots — inventory + verification (pending / in_review / verified / rejected / na); plot sale states available/reserved/sold.
  • Properties — seller uploads, approvals, edits; admins add one-off listings (/admin/property/new) or bulk-import /admin/property/import (.xlsx template + sample row; staff skip Add/Import).
  • Appointments — site visit roster (pending, confirmed, cancelled, completed…).
  • Requests — buyer land-request pipeline (open/in_progress/matched/closed).
  • Flags — feature-flag JSON surfaced through /api/feature-flags.
  • Services — /api/service-booking payloads for all 6 Phase II field services (fencing, borewell, drip, farming plan, plants, fertilizers/nutrition).
  • Feedback — business feedback submitted via /feedback public page (or POST /api/feedback). Fields: business name, contact, type, message, rating. Status: new → read → replied → archived. Admin adds internal notes.
  • Testimonials — ops create/edit/approve testimonial cards and win stats with optional photo upload. Approved entries appear on the homepage (cached 2 min).

Exact controls evolve — treat the deployed UI + Supabase rows as authoritative.

After login — default home

  • Buyer (default role) → /dashboard
  • Seller → /seller (may also open /dashboard for buyer flows)
  • Admin / staff → /admin

Mirrors homePathForRole on login + OAuth callback; ?next= wins when it is a safe internal path. Navbar / header brand links go to the hub when a session exists (not the brochure /).
